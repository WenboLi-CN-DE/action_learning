# Feature Landscape

**Domain:** 企业信息透明化平台（打通销售/咨询与研发/产品）
**Researched:** 2026-04-24
**Overall confidence:** MEDIUM（基于竞品分析+生态调研，但本项目是管培生行动学习课题而非商业产品，部分商业平台特征需裁剪）

---

## Table Stakes

用户期望必须有的功能。缺失 = 产品感觉不完整、用户不愿使用。

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| **项目状态可视化面板** | 后端研发做了什么、进度如何——这是"让前端看到后端在做什么"的核心承诺，缺失则平台无意义 | Low | 简单列表/卡片视图即可，初期数据量少（5-20条），不需要复杂筛选。参考 Productboard 的 feature status tracking、Jama Connect 的 real-time visibility |
| **客户需求提交表单** | "让后端能看到客户需要什么"——前端必须有提交需求的入口，否则信息流断裂 | Low | 基础表单（标题+描述+行业标签+提交人），不需要复杂工作流。参考 ProductLift 的 guest submissions、Productboard 的 feedback portal |
| **需求-项目关联/匹配** | 平台核心价值是"可关联"——如果需求和项目是两条平行线，信息透明只是两块独立的看板，用户会问"那我为什么要用一个平台" | Medium | 初期手动关联（选择已有项目链接到需求）即可。自动匹配（基于标签相似度）是后续增强。参考 Arkweaver 的 feature-request mapping、Mythrilite 的 engineering-to-pipeline mapping |
| **行业/业务线标签分类** | 公司多业务线并行（数据中心、水处理、工业自动化），没有分类则数据无序、信息变成噪声 | Low | 固定标签集 + 自定义标签。初期只用行业维度，不需要多层级分类体系。参考 Aha! 的 categorization、Productboard 的 segmentation |
| **全透明信息展示** | 5个角色看到完全一样的信息——这是MVP的显性设计决策，缺失等于回到各自的信息孤岛 | Low | 无权限区分是最简实现。但需注意：全透明 ≠ 无结构，信息仍需分类组织。参考 HBR《Transparency Trap》和 Paradigm Red 的研究：透明需要配合"渐进式展示"而非一股脑堆砌 |
| **基础搜索** | 数据虽少，但用户仍需快速找到特定项目/需求。没有搜索则用户只能翻列表 | Low | 初期简单文本搜索即可（标题+描述）。不需要 Elasticsearch 级别，SQLite FTS5 足够 |
| **操作人/提交人标识** | 信息可追溯的前提——谁提交了什么需求、谁更新了项目状态 | Low | 自动记录提交人和时间戳。不需要完整的审计日志系统 |
| **变更时间线/历史记录** | "可追溯"承诺——项目状态变了什么、何时变，否则用户对信息可信度存疑 | Medium | 简单版本：显示最近更新时间和更新人。完整版本：变更历史列表。MVP 先做简单版 |

## Differentiators

让平台脱颖而出、超出预期但用户认可的功能。不是必选项，但有了会让产品明显更好。

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| **匹配度可视化** | 直接告诉用户"这个需求被多少项目覆盖""这个项目覆盖了多少需求"——让信息关联可量化，不再是模糊感觉 | Medium | 覆盖率热力图/百分比。初期可在项目卡片上显示"覆盖需求数"，需求卡片上显示"匹配项目数"。参考 Mythrilite 的 revenue-pipeline mapping（将工程投入映射到业务信号） |
| **需求反馈闭环通知** | 当后端项目状态更新影响到某个需求时，让提交该需求的前端同事知道——这才是"信息双向透明"的完整闭环 | Medium | MVP 不做实时推送（已明确 Out of Scope），但可以做"手动刷新后高亮变更项"。低成本实现：标记上次查看后有哪些项目/需求发生了状态变更 |
| **团队协作记录（评论/讨论）** | 前后端就某个需求/项目的沟通记录和决策存档——让"为什么这么做"可见，而非只看到结果 | Medium | 每个项目/需求下的评论区。异步文本讨论即可。参考 Aligni 的 DiscussAnything™（将讨论绑定到数据对象）、Slack 的 threaded discussions |
| **标签驱动的过滤视图** | 按"数据中心""水处理"等标签过滤看板——让多业务线用户只关注自己的领域，降低认知负荷 | Low | 下拉选择标签 → 过滤列表。关键洞察：全透明 + 过滤 = 渐进式透明（解决"全透明导致信息过载"问题）。参考 Steven Brough 的研究：核心信号保持可见，深层细节需要主动选择才展示 |
| **项目优先级/紧急度标记** | 让前端了解后端的优先判断——"研发为什么先做这个项目而不是那个需求" | Low | 简单优先级标签（高/中/低）或排序。参考 Productboard 的 prioritization scoring、Aha! 的 feature ranking |
| **数据导出** | 用户可能需要将信息导出到 Excel/PDF 呈报给管理层 | Low | CSV 导出即可，不需要复杂报表。管培生课题成果展示也需要导出功能 |

## Anti-Features

**刻意不做的功能**——做了反而伤害产品或分散资源。

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| **实时推送通知** | 已明确 Out of Scope。初期数据量少、更新频率低，推送反而增加开发复杂度（websocket/长连接）且无实际价值。手动刷新 + 变更高亮即可 | 变更高亮标记（上次查看后的变更项醒目显示） |
| **角色权限区分/RBAC** | MVP 全透明是显性设计决策。加权限 = 增加开发量 + 违背透明承诺 + 引入管理复杂度。权限系统是"看似简单实则复杂"的典型陷阱（用户模型、权限矩阵、测试矩阵） | 无权限区分。所有角色同视图。后续按需添加 |
| **对接CRM/项目管理工具** | 公司现有系统对接需要了解具体API、权限、数据模型——管培生团队无此能力且时间有限。对接失败率高，且初期数据量少不值得自动化 | 手动录入 + 预留接口设计空间（API schema设计好但暂不实现） |
| **实时协作编辑** | Google Docs 级别的多人同时编辑是技术复杂度的黑洞（CRDT/OT算法）。管培生团队无能力实现，且使用场景不明确 | 异步评论 + 独立编辑。每个人编辑自己提交的内容 |
| **复杂数据分析仪表盘** | BI级仪表盘（图表、趋势线、KPI追踪）看起来专业但初期只有5-20条数据——图表比数据还多，沦为装饰 | 简单计数/统计（"数据中心业务线有X个项目、Y个需求"）。后续数据量增长时再加图表 |
| **AI驱动的自动匹配/推荐** | AI匹配需要足够数据量和训练，初期5-20条数据AI无意义。且增加技术栈复杂度（向量搜索/ML模型） | 手动关联 + 标签相似度提示（"这两个项目/需求有相同行业标签，是否关联？"） |
| **移动端App** | 已明确 Web 优先。移动端 = 双平台开发/测试 + App分发渠道问题（内网环境） | 移动端友好的Web响应式布局 |
| **社交化功能（点赞/投票/排名）** | Productboard/Aha! 的投票功能适合外部用户社区，但本项目是内部5-20人团队——投票排名在此规模下是表演性行为而非实质决策 | 需求优先级由提交人/团队讨论决定，标注优先级标签即可 |
| **工作流自动化** | 自动审批流、状态自动推进等——初期流程未定型，自动化固化了不确定的流程 | 手动状态更新。流程成熟后再自动化 |
| **全文知识库/Wiki** | Wiki式知识库（Notion/Confluence级别）是另一个产品。本项目聚焦"项目-需求-匹配"，知识管理是不同领域 | 项目/需求描述字段够长即可，不单独做Wiki模块 |

## Feature Dependencies

功能间的依赖关系，决定构建顺序。

```
行业标签分类 ─────┐
                  ├──→ 标签过滤视图（依赖标签系统）
                  │
项目状态面板 ──────┤
                  ├──→ 需求-项目关联（需要项目和需求两个实体都存在）
客户需求表单 ──────┤
                  │
                  ├──→ 匹配度可视化（依赖关联关系）
                  │
评论/协作记录 ────→ 依赖项目/需求实体（评论挂在实体上）

基础搜索 ────────→ 依赖数据存在

变更高亮 ────────→ 依赖变更时间线记录

数据导出 ────────→ 依赖核心数据模型完成
```

**关键依赖链：**
1. **基础实体先行**：项目 + 需求 + 标签 → 这是所有后续功能的地基
2. **关联是核心桥**：需求-项目关联 → 匹配度可视化 → 这才是"双向透明"的实质
3. **协作是增值层**：评论/讨论 → 依赖基础实体和关联，但不是 MVP 必须
4. **过滤是体验层**：标签过滤 → 依赖标签系统，但实现成本低，可同步推出

## MVP Recommendation

基于"管培生行动学习课题 + 5-20条数据 + 非专职开发团队 + Python后端 + React前端"的约束：

### 必须实现（Phase 1）

1. **项目状态可视化面板** — 核心承诺，无此则平台无意义
2. **客户需求提交表单** — 信息流另一端，必须可录入
3. **行业/业务线标签分类** — 多业务线场景的基础组织维度
4. **需求-项目手动关联** — 双向透明的实质，项目-需求不再是两条平行线
5. **全透明同视图展示** — 设计决策已定，也是最简实现
6. **基础搜索** — 低成本但体验关键
7. **操作人/时间戳** — 可追溯性的最低要求

### 推荐同步实现（Phase 1 增值）

8. **标签过滤视图** — 低复杂度但解决"全透明信息过载"问题，与标签系统一起做
9. **变更高亮标记** — 低成本替代推送通知，解决"我怎么知道什么变了"的问题

### Phase 2 增值

10. **匹配度可视化** — 让关联可量化（"覆盖3个需求""匹配2个项目"）
11. **评论/协作记录** — 决策存档和沟通记录
12. **数据导出** — 管培生成果展示需要

### 明确推迟

- **需求反馈闭环通知**：Phase 2，因为依赖变更追踪机制成熟
- **项目优先级标记**：Phase 2，因为优先级体系需团队讨论确定
- **标签相似度提示**：Phase 3，在手动关联经验积累后考虑半自动化

## Competitive Landscape Reference

本项目虽是内部行动学习课题而非商业产品，但以下商业平台的特征谱系可作为参照：

| Platform | Core Focus | Relevant Feature | Our Applicability |
|----------|-----------|-----------------|-------------------|
| **Productboard** | 客户反馈→产品优先级→路线图 | 需求收集+优先级评分+反馈闭环 | 需求表单+关联映射（裁剪评分系统） |
| **Aha!** | 战略规划+路线图+想法管理 | 多层级分类+想法门户+战略对齐 | 行业标签分类+需求入口（裁剪战略层） |
| **Mythrilite** | 工程投入→业务信号→ROI映射 | 项目-管道收入映射+容量可视化 | 需求-项目匹配度可视化（裁剪财务维度） |
| **Arkweaver** | 特性需求→路线图→收入关联 | 特性-客户-收入三角映射 | 需求-项目-标签三角关联（裁剪自动化） |
| **Aligni** | 上下文协作+讨论绑定数据 | DiscussAnything™（评论绑定到数据对象） | 项目/需求下的评论（简化版） |
| **Propel (PLM)** | 销售-工程对齐+单一产品线 | 销售-工程共享平台+配置/报价 | 双向信息面板（裁剪报价/配置维度） |
| **Provarity** | 预销售流程管理 | 评估跟踪+信号收集+自动状态摘要 | 项目状态面板+变更追踪（裁剪评估流程） |

**关键洞察：** 这些商业平台都面向特定行业场景（SaaS产品管理、制造业PLM、预销售流程），功能深度远超本项目需要。本项目的独特性在于：
- 不是商业产品而是行动学习课题，用户是5-20人管培生团队
- 核心价值不是"管理流程"而是"信息可见"
- 数据量极小（5-20条），复杂筛选/分析无意义
- 全透明无权限是刻意设计而非功能缺失
